# LLM и агенты на практике: как это применить к AdventAI

> Практический гайд. Основан на конспекте лекции 2-й недели AI Advent Challenge (контекст, память,
> планирование, инструменты, токены), но переложен на **реальный код AdventAI**: для каждого концепта —
> «суть → как сейчас в проекте → что конкретно сделать», со ссылками на файлы.

## 0. Как устроен агент сейчас

Поток одного сообщения:

```
ChatScreen → ChatViewModel.sendMessage()
          → AiAgent.ask(config, userRequest: String)        // core/domain/.../agent/AiAgent.kt
          → AiChatRepository.sendDetailedMessage(message)    // core/data/.../repository/AiChatRepositoryImpl.kt
          → ChatMapper.toChatRequestDto / toMessageDtos      // core/data/.../mapper/ChatMapper.kt
          → DeepSeekApi / OpenRouterApi  (POST chat/completions)
          → LlmAnswer → AgentAnswer → UI
```

**Статус (Day 7, сделано ✅):** история диалога теперь **персистится в SQLite (Room)** и **целиком
отправляется в LLM**. Источник истины для сообщений — Room (`ChatHistoryRepository` →
`ChatMessageDao`), `ChatViewModel` подписан на `observeMessages(agentId)`. `AiAgent.ask(config,
conversation)` принимает всю историю, `ChatMapper` разворачивает её в `messages` с ролями. Агент помнит
контекст между запусками приложения. Осталось из памяти/контекста: sliding window, history compression,
долговременная память по фактам (разделы 1–2).

---

## 1. Контекст-менеджмент → AdventAI

Суть: модель не помнит ничего сама — всё нужное передаётся в `messages` каждого запроса. Качество ответа = качество переданного контекста.

**Как сейчас:** контекст = `system prompt` (роль + тема диалога) + 1 сообщение. Истории нет.

**Что сделать (по возрастанию сложности):**

1. **Передавать историю диалога** — базовая многоходовость. Минимальные правки:
   - `AiAgent.ask(config, history: List<AgentChatMessage>)` вместо одной строки;
   - `AiChatRepository.sendDetailedMessage(messages: List<ChatMessage>)`;
   - `ChatMapper.toMessageDtos` уже умеет роли (`system`/`user`) — добавить ветку для `assistant` и
     разворачивать историю в список `MessageDto`.
2. **Sliding window** — когда история длинная, слать только последние N сообщений (+ системный промпт).
   Простая защита от роста токенов; хорошо работает для чата на одну тему (как «Визовый специалист»).
3. **History compression** — при приближении к лимиту окна свернуть старую часть истории в summary
   (отдельным запросом к дешёвой модели) и подставлять свёртку в системный промпт. Делать **редко** —
   суммаризация теряет детали.
4. **Пересоздание диалога / `clear`** — кнопка «Новый диалог», которая сбрасывает `messages`. Дёшево и
   эффективно: для новой темы лучше начать с чистого окна, чем тащить старый шум.

> `context layering`, `topic branches`, `vector embeddings` — для текущего одно-агентного чата избыточны.
> Вернуться к ним, когда появятся несколько агентов с общим знанием или RAG.

## 2. Управление памятью → AdventAI

Суть: память бывает session (в пределах сессии) и long-term (между сессиями). Модель «узнаёт» пользователя только если ему это подгрузили.

**Как сейчас:**
- **Session memory** = `ChatViewModel._uiState.messages` — живёт в памяти, теряется при смерти процесса и
  при уходе с экрана (ViewModel привязан к nav-entry чата). Долговременной памяти нет вообще.

**Что сделать:**

1. **Персист диалога** (session → durable): сохранять сообщения в Room/DataStore по `agentId`, чтобы чат
   восстанавливался. Это и есть «session memory, которую не теряем».
2. **Long-term memory по агенту** (когда появится): хранить факты о пользователе (например, гражданство,
   страна назначения для визового агента) и подмешивать в системный промпт. Начать с самого простого —
   **compressed memory** (короткое summary на агента), это мало токенов и уже даёт эффект «помнит меня».
3. Более продвинутое — `vector-based` / `self-reflective` память — оставить на потом (нужны embeddings и
   отдельная модель). Важно из лекции: память хранит факты, но не гарантирует верных выводов — не
   доверять ей слепо.

## 3. Контекстное окно и токены → AdventAI

Суть: окно не бесконечно (ориентир ~1M токенов у топ-моделей). При оплате per-token каждый лишний токен — деньги. DeepSeek в проекте — как раз платная per-token модель.

**Как сейчас (есть что улучшить):**
- **Reasoning тратится впустую.** Модели `deepseek-v4-flash/pro` всегда возвращают `reasoning_content`
  (проверено логом). В коде есть `ThinkingMode.Disabled` (`ChatRequestOptions.kt`) и поддержка поля
  `thinking`, но `AiAgent.toRequestOptions()` ставит `thinkingMode = null` → рассуждение **не
  отключается** ни для одного запроса. Для быстрых/коротких ответов это лишние токены и задержка.
- **`maxTokens = null`** → ответ ничем не ограничен.
- **Токены уже считаются, но выбрасываются.** `ChatMapper` собирает `LlmAnswer.usage` (`TokenUsage`,
  включая cache hit/miss), но `AiAgent` маппит в `AgentAnswer(content, modelTitle)` и **теряет `usage` и
  `reasoningContent`**.

**Что сделать:**
1. Прокинуть `thinkingMode` в `AgentConfig`/`AiAgent` и давать пользователю/агенту выбор «быстро без
   reasoning» vs «думать». Для простых вопросов — `Disabled`.
2. Задать разумный `maxTokens` по умолчанию (защита от мегаответов и от перерасхода).
3. Не выбрасывать `usage`: пробросить в `AgentAnswer` и показывать в настройках/диагностике (видно
   стоимость запроса). `promptCacheHitTokens` подскажет, работает ли кэш промпта.

## 4. Планирование → AdventAI

Суть: для многошаговых задач агент сам определяет цель → дробит на задачи → выбирает стратегию → (под)тверждает план → исполняет (как Planning Mode в Claude Code).

**Как сейчас:** агент одно-ходовый (вопрос → ответ), планирование не нужно.

**Когда понадобится:** как только агент получит инструменты или будет вести многошаговый сценарий
(например, «собери чек-лист документов и проверь сроки по стране»). Тогда — отдельный шаг «составить
план» перед исполнением. Пока держать в уме, не усложнять.

---

## 5. Оркестрация инструментов → AdventAI

Суть: tool selection → invocation → получение результата → интерпретация → обработка ошибок → обновление состояния. Тема недели MCP.

**Как сейчас:** инструментов нет — есть только вызов LLM.

**Что подготовить заранее:**
- Архитектура уже удачная: `AiAgent` — единая точка, через которую можно добавить tool-calling, не
  трогая UI. Repository-слой изолирует HTTP.
- Когда дойдёшь до инструментов: возвращать из LLM не только текст, а структурированный tool-call,
  исполнять его (поиск по визовым правилам, калькулятор сроков и т.п.), результат класть обратно в
  `messages` и звать модель ещё раз. Обязательно обрабатывать ошибки инструмента (API/таймаут) — в коде
  уже есть хороший базис в `AiChatRepositoryImpl` (маппинг ошибок в `AppError`).

## 6. Промпт-инжиниринг → AdventAI

Суть: роль + цель + стек + ограничения + формат + примеры сужают «бесконечный» контекст знаний и резко поднимают качество. Уровни: simple → pro → pro-max (мета-промптинг).

**Как сейчас (уже неплохо):** `AiAgent.buildSystemPrompt()` собирает роль агента + тему диалога +
инструкцию «отвечать практично». Это уже уровень выше «simple».

**Что улучшить:**
- Добавить в системный промпт **формат ответа** (например, для визового агента: «структура: тип визы →
  документы → сроки → риски → следующий шаг») и **ограничения** (не выдумывать сроки, помечать, что
  нужно сверять с офиц. источником — частично уже есть).
- Хранить эти расширенные промпты как пресеты агента (`AgentConfig.systemPrompt`) — пользователь может
  редактировать их в bottom sheet настроек.
- `pro-max`/мета-промптинг: для «тяжёлых» агентов можно генерировать подробный системный промпт один раз
  и сохранять — дороже на старте, но экономит уточняющие шаги.

---

## 7. JSON vs TOON → AdventAI

Суть: JSON удобен, но «ест» токены за счёт повторения имён полей; TOON компактнее при оплате per-token.

**Как сейчас:** запрос/ответ — обычный JSON через Gson. Для чата (текст в `content`) разница невелика.

**Что сделать:** пока ничего. Вспомнить про TOON/компактные форматы, только если появятся запросы с
большими **структурированными** данными (массивы объектов в tool-calling/RAG) и счёт за токены станет
заметным.

## 8. Приоритетный план улучшений (для AdventAI)

От «дёшево и сразу заметно» к «на будущее»:

1. ✅ **(Day 7) Передавать историю диалога в запрос** (раздел 1.1) — сделано.
   Файлы: `AiAgent.kt`, `AiChatRepository(+Impl)`, `ChatMapper.kt`, `ChatViewModel.kt`.
2. **Sliding window + кнопка «Новый диалог»** (1.2, 1.4) — контроль токенов и чистый старт темы.
   `ChatViewModel.onClearDialog()` уже есть — осталось вывести кнопку в UI.
3. **Управление reasoning + `maxTokens`** (раздел 3.1–3.2) — скорость и экономия токенов.
4. **Не терять `usage`/`reasoningContent`** (3.3) — показывать стоимость и (опц.) ход рассуждения.
5. ✅ **(Day 7) Персист диалога (Room)** (раздел 2.1) — сделано: `ChatHistoryRepository` + `ChatMessageDao`,
   история переживает перезапуск.
6. **Расширенные системные промпты с форматом ответа** (раздел 6).
7. Долговременная память по агенту (2.2), затем — инструменты/планирование (4–5).

> Принцип из лекции, который держим в голове: **сужать контекст и не доводить диалог до постоянной
> суммаризации**. Лучше явные роли, история по делу и чистый старт темы, чем «болото» из всего подряд.
