# Day 9 — Управление контекстом: сжатие истории

Задание AI Advent Challenge #8, день 9. Ветка `task8` (поверх Day 8 «Токены»).

## Задание

Реализовать управление контекстом через сжатие истории:
- хранить последние N сообщений «как есть»;
- остальное заменять summary (свёртка пачками, «каждые 10 сообщений»);
- хранить summary отдельно и подставлять его в запрос вместо полной истории.

Сравнить: качество ответов без сжатия vs со сжатием и расход токенов до/после.
Результат — агент, который работает с компрессией истории и экономит токены.

## Идея (rolling summary)

```
Контекст запроса = system prompt
                 + [СЖАТАЯ ПАМЯТЬ: summary старой части диалога]   ← отдельный блок
                 + последние N сообщений «как есть»
```

- **Последние N сырых** сообщений (`keepRecentMessages`, дефолт 6) всегда уходят как есть.
- **Старшее сверх N** сворачивается в `summary` пачками (`summarizeBatch`, дефолт 10 — «каждые 10»).
- Свёртка **инкрементальная (rolling)**: `новый summary = LLM(старый summary + очередная пачка
  старых сообщений)`. Не пересуммируем всё заново — это держит и стоимость, и размер summary
  ограниченными.
- Суммаризатор работает на дешёвой **DeepSeek V4 Flash** с выключенным reasoning и маленьким
  `maxTokens`. Summary **структурированный** — два раздела: `ФАКТЫ` (имя/числа/решения — при каждой
  пересборке копируются **дословно** и дополняются) и `КОНТЕКСТ` (сжимаемый пересказ). «Прибитые»
  факты не размываются при rolling-свёртке — за счёт этого имя из Day 8 «выживает», качество не падает.
- Состояние на диалог — всего два поля: `summary` и `summarizedCount` (сколько старых сообщений уже
  свёрнуто). Сообщения append-only, поэтому счётчика достаточно.

## Что сделано

### 1. Ядро сжатия (домен)

- `HistoryCompressor` — чистая логика разбиения истории на `toSummarize` / `recent` по
  `(summarizedCount, keepRecent, batch)`. Покрыта юнит-тестом `HistoryCompressorTest`.
- `ConversationSummarizer` — rolling-свёртка через `SendDetailedChatMessageUseCase` (DeepSeek Flash,
  `ThinkingMode.Disabled`, `maxTokens ≈ 400`), возвращает обновлённый summary + `usage` (токены самой
  свёртки — для честного учёта «доплаты»).
- `AiAgent.ask(config, conversation, memory)` — `memory` (summary) вставляется отдельным блоком в
  system prompt; `conversation` тогда = только сырой хвост.
- `PrepareCompressedContextUseCase` — оркестрация для чата: грузит состояние из Room, при
  необходимости сворачивает, сохраняет новый summary и возвращает `memory + recent`.

### 2. Хранение summary отдельно (Room, v5)

- `ConversationEntity` += `summary`, `summarized_count` — summary живёт на строке диалога, **отдельно**
  от самих сообщений (`chat_messages`).
- `ChatHistoryRepository.getSummaryState / updateSummary`. Версия БД поднята до 5
  (`fallbackToDestructiveMigration`, dev — миграция не пишется).

### 3. Сжатие в реальном чате

- В настройках агента (⚙ → «Сжатие истории») — тумблер **«Сжимать историю в summary»** + чипы N и
  размера пачки. Когда включено, `ChatViewModel.sendMessage` гоняет историю через
  `PrepareCompressedContextUseCase` и шлёт `summary + последние N` вместо полной истории.
- В ленте — карточка **«Сжатая память»**: сколько сообщений свёрнуто и сам текст summary (разворот по
  тапу). История на экране остаётся полной — сжимается только то, что уходит в модель.
- В индикаторе окна — строка экономии: **«сжато: −P% (полная ~L ток.)»** (оценка `TokenEstimator`:
  сжатый контекст vs полная история, пересчитывается вживую).

### 4. A/B-экран сравнения (сжатие vs без)

⚙ → «Сжатие истории» → **«Сравнить: сжатие vs без»**. Один и тот же сценарий (имя → серия «тяжёлых»
визовых вопросов → проверка «как меня зовут?») прогоняется дважды — **без сжатия** (вся история) и
**со сжатием** (`summary + последние N`), и показывается богатая статистика:

- **Таблица сравнения**: вход / выход / всего токенов, накладные на свёртки, **пик контекста**, из
  кэша, **стоимость с кэшем и без кэша**, «вспомнил имя» + сам финальный ответ модели.
- **График размера контекста по ходам** (2 линии): без сжатия контекст растёт линейно, со сжатием —
  плоский. Наглядно видно, что именно экономит сжатие.
- **Таймлайн свёрток**: какие сообщения свернулись, получившийся `summary` (раскрывается), его размер,
  бейдж «хранится отдельно» — видно сам процесс и что summary — отдельная сущность.
- **Итог** с пояснением про кэш (см. ниже).

Эксперимент изолированный — история живёт в памяти прогона, в Room ничего не пишется.

## Поток работы агента

```
ChatScreen → ChatViewModel.sendMessage()
          → PrepareCompressedContextUseCase            // summary + последние N (свёртка при необходимости)
              → HistoryCompressor.plan(...)            // что свернуть / что оставить
              → ConversationSummarizer.summarize(...)  // отдельный запрос к DeepSeek Flash
              → ChatHistoryRepository.updateSummary(...) // summary в Room (отдельно от сообщений)
          → AiAgent.ask(config, recent, memory = summary)
          → AiChatRepository → DeepSeekApi / OpenRouterApi
          → AgentAnswer(content, usage) → история и токены в Room → UI
```

- `core/domain/.../agent/ConversationSummarizer.kt` — rolling-свёртка через LLM.
- `core/domain/.../agent/AiAgent.kt` — параметр `memory`, оценка сжатого контекста.

## Как проверить / снять видео

1. **Реальный чат.** ⚙ → «Сжатие истории» → вкл, N=6, пачка=10. Назовите имя, задайте 12–15 вопросов:
   когда наберётся пачка, появится карточка «Сжатая память», индикатор покажет «сжато: −P%». Спросите
   «как меня зовут?» — агент вспомнит (имя в summary), хотя ранних сообщений в запросе уже нет.
2. **A/B-экран.** ⚙ → «Сравнить: сжатие vs без» → «Запустить сравнение». Дождитесь обоих прогонов:
   у «со сжатием» вход и всего токенов заметно ниже, имя на месте — итоговая карточка покажет −P% и
   «качество сохранено».

## Заметки

- Оценщик токенов в индикаторе — **эвристика** (`TokenEstimator`), а не BPE; точные цифры берём из
  `usage` API (они и считаются на A/B-экране и в «Статистике»).
- Свёртка тратит токены сама по себе — на A/B-экране они показаны отдельной строкой «на свёртки» и
  включены в «всего токенов», чтобы сравнение было честным.
- **Важно про деньги и кэш промпта.** У DeepSeek переотправка одной и той же истории почти целиком
  попадает в **кэш промпта** (вход дешевле в ~4×: $0.07 vs $0.27 за 1M). Сжатие меняет контекст
  каждый ход и **ломает кэш**, плюс добавляет платные вызовы суммаризации — поэтому на коротком
  диалоге «со сжатием» может выйти **дороже по деньгам**, хотя **сырых токенов и размера контекста
  меньше**. Реальная денежная польза сжатия — на моделях без кэша, на очень длинных диалогах и когда
  полная история **не влезает в окно**. Поэтому A/B показывает стоимость **и с кэшем, и без кэша**.
- Summary хранится **отдельно** от сообщений: история на экране всегда полная, сжимается только то,
  что реально уходит в модель.
- Room пересоздаётся через `fallbackToDestructiveMigration` (dev), версия схемы поднята до 5.
